# Thinking Depth

Deep analysis is the default for non-trivial work. These methods apply to any task requiring evaluation, comparison, debugging, architectural decisions, or recommendations.

## Self-Refine Loop

After completing any non-trivial analysis or recommendation:
1. Critique your own output — what did you miss? What assumptions did you make? Are examples concrete enough?
2. List 3 questions your analysis doesn't answer
3. Try to answer them using available tools
4. Revise your output based on what you found
5. Only finalize after at least one critique-refine cycle

## Metacognitive Check

Before finalizing any recommendation or conclusion:
1. Restate the problem — what is actually being evaluated?
2. State your initial judgment
3. List assumptions you haven't verified
4. Identify evidence that would change your conclusion
5. Is there a simpler explanation for what you're seeing?
6. State final judgment with confidence level

## Hypothesis-Driven Analysis

When analyzing unfamiliar code, patterns, or systems:
1. Generate hypotheses before investigating — "I expect X because of Y"
2. Test each hypothesis with available tools
3. Record surprises — contradictions are more valuable than confirmations
4. Rank findings with surprises and contradictions first

## Socratic Self-Questioning

Before concluding on any non-trivial question:
- "What evidence would change my mind?"
- "What am I assuming that I haven't verified?"
- "Is there a simpler explanation?"
- "Would someone who disagrees have a valid point?"

## When to Apply

- Always for Tier 3 tasks (see 02-research-depth.md)
- Any task involving pattern evaluation or competing approaches
- When debugging complex issues
- When making architectural or convention recommendations
- NOT for Tier 1 instant answers or simple questions
